# Variable and Function Proposal

## Variables

### Canonical Struct

Variable and function identifiers are exported together through a parcel-level struct. The struct tag and the variable of that type share a canonical name derived from the export path and parcel name, with each `/` replaced by `_`:

```
<path>/<name>  →  <path_underscored>_<name>
```

For example, path `foo` with parcel name `p` produces the canonical name `foo_p`.

### Export

For each variable identifier `<id>` of type `<T>` in the parcel interface, the translator generates a pointer-typed member in the canonical struct and initialises it with the variable's address in `export/<path>/<name>`:

```c
struct <canonical> { <T> *<id>; } <canonical> = { &<id> };
```

The export file must be included _after_ all variable definitions it references.

### Import

The import file `import/<path>/<name>.<stem>` re-declares the struct type, forward-declares the canonical variable as `extern`, and defines a `<stem>`-named pointer to it:

```c
struct <canonical> { <T> *<id>; };
extern struct <canonical> <canonical>;
struct <canonical> *<stem> = &<canonical>;
```

The importer reads a variable as `*<stem>-><id>` and writes to it as `*<stem>-><id> = <value>`.

### Example

**`foo.c`** — declares and exports parcel `p` at path `foo`, exporting variable `v`:

```c
#pragma parcel p { v }

int v = 0;

#include "export/foo/p"
```

**`_parcel/export/foo/p`** — generated by the translator:

```c
struct foo_p { int *v; } foo_p = { &v };
```

**`bar.c`** — imports parcel `foo/p` with stem `s`:

```c
#include "import/foo/p.s"

int x = *s->v;
*s->v = 42;
```

**`_parcel/import/foo/p.s`** — generated by the translator:

```c
struct foo_p { int *v; };
extern struct foo_p foo_p;
struct foo_p *s = &foo_p;
```

`*s->v` reads the current value of `v` defined in `foo.c`; assigning to `*s->v` modifies it through the pointer.

## Functions

### Export

For each function identifier `<f>` returning `<R>` with parameter list `<P>`, the translator generates a function-pointer member in the canonical struct and initialises it with the function's address in `export/<path>/<name>`:

```c
struct <canonical> { <R> (*<f>)(<P>); } <canonical> = { <f> };
```

The canonical struct name follows the same scheme as for variables. Where a parcel exports both variables and functions, all members appear in a single struct.

### Import

The import file follows the same form as for variables. The importer calls the function as `<stem>-><f>(<args>)`.

### Example

**`foo.c`** — declares and exports parcel `p` at path `foo`, exporting function `f`:

```c
#pragma parcel p { f }

void f(int x) { /* ... */ }

#include "export/foo/p"
```

**`_parcel/export/foo/p`** — generated by the translator:

```c
struct foo_p { void (*f)(int); } foo_p = { f };
```

**`bar.c`** — imports parcel `foo/p` with stem `s`:

```c
#include "import/foo/p.s"

s->f(42);
```

**`_parcel/import/foo/p.s`** — generated by the translator:

```c
struct foo_p { void (*f)(int); };
extern struct foo_p foo_p;
struct foo_p *s = &foo_p;
```

`s->f(42)` in `bar.c` calls `f` as defined in `foo.c` through the function pointer.

## Criticisms

**1. Struct definition duplicated across export and import files.**
The struct type is defined independently in both generated files. If a translation unit ever includes both — possible in a selector header that both re-exports and imports a parcel, or during refactoring — the duplicate struct definition is a constraint violation in C (even if the members are identical). Include guards on the export file, or a separate generated canonical header for the struct type, would be needed to handle this correctly.

**2. Variable access syntax is asymmetric with the README.**
The README states that variables and functions use the `stem->identifier` form. Under this proposal, variables require `*stem->id` (pointer dereference) while functions use `stem->f(args)` (direct call through function pointer). These are different access patterns for the two cases. The README's uniform `stem->identifier` description does not hold.

**3. Function-pointer variables get an extra level of indirection.**
The `hello_world` example exports `output`, which is a variable of function-pointer type (`out_Output`). As a variable, this proposal makes its struct member `out_Output *output` (pointer to function pointer), requiring `(*s->output)(greeting)` to call — not `s->output(greeting)` as the README shows. The proposed scheme does not special-case function-pointer typed variables, producing usage that is inconsistent with the documented `stem->identifier` syntax and with the existing example.

**4. Exported identifiers must have static storage duration.**
The struct initialiser uses `&id` for variables and `f` for functions. Both are valid C static initialisers only when the referenced identifiers have static storage duration (file scope). Exporting a local or `auto` variable would produce a constraint violation. The translator should diagnose this, but no such diagnostic is listed in `TRANSLATION.md`.

**5. No support for `const`-qualified exports.**
All variable members are `T *` (non-const pointer). There is no mechanism for a parcel to export a read-only variable. A `const` variable (`const int v = 0`) exported via `int *v` in the struct loses the `const` qualifier, potentially misleading importers and silencing compiler warnings about modification through the pointer.
