# Variable and Function Proposal: Nested Struct

## Variables

### Canonical Nesting

Variable and function identifiers are exported through a hierarchy of nested structs that mirrors the segments of the export path. The first path segment names both a global struct tag and a global struct variable. Each subsequent path segment names a member of the enclosing struct, itself a struct. The nesting terminates at the parcel-named struct, whose members hold pointers to the exported identifiers.

Struct tags at each level are formed by joining all path segments up to and including that level with `_`. The innermost tag additionally incorporates the parcel name:

```
path: a/b   name: p

struct a {                  ← tag: a,   variable: a
    struct a_b {            ← tag: a_b
        struct a_b_p {      ← tag: a_b_p  (= underscore canonical name)
            <T> *<id>;
        } p;
    } b;
} a;
```

For the common single-segment path `foo`, parcel `p`, this reduces to:

```
struct foo {                ← tag: foo,  variable: foo
    struct foo_p {          ← tag: foo_p
        <T> *<id>;
    } p;
} foo;
```

The stem pointer in the import file points to the innermost struct: `&foo.p` (type `struct foo_p *`). The innermost struct tag is identical to the underscore canonical name of the preceding proposal.

### Export

For each variable identifier `<id>` of type `<T>` in the parcel interface, the translator generates the innermost struct type, the enclosing path struct, and an initialiser in `export/<path>/<name>` (shown here for a single-segment path):

```c
struct <path>_<name> { <T> *<id>; };
struct <path> { struct <path>_<name> <name>; } <path> = { .<name> = { &<id> } };
```

The export file must be included _after_ all variable definitions it references.

### Import

The import file `import/<path>/<name>.<stem>` re-declares both struct types, forward-declares the global struct variable as `extern`, and defines a `<stem>`-named pointer to the innermost struct:

```c
struct <path>_<name> { <T> *<id>; };
extern struct <path> { struct <path>_<name> <name>; } <path>;
struct <path>_<name> *<stem> = &<path>.<name>;
```

The importer reads a variable as `*<stem>-><id>` and writes to it as `*<stem>-><id> = <value>`.

### Example

**`foo.c`** — declares and exports parcel `p` at path `foo`, exporting variable `v`:

```c
#pragma parcel p { v }

int v = 0;

#include "export/foo/p"
```

**`_parcel/export/foo/p`** — generated by the translator:

```c
struct foo_p { int *v; };
struct foo { struct foo_p p; } foo = { .p = { &v } };
```

**`bar.c`** — imports parcel `foo/p` with stem `s`:

```c
#include "import/foo/p.s"

int x = *s->v;
*s->v = 42;
```

**`_parcel/import/foo/p.s`** — generated by the translator:

```c
struct foo_p { int *v; };
extern struct foo { struct foo_p p; } foo;
struct foo_p *s = &foo.p;
```

`*s->v` reads the current value of `v` defined in `foo.c`; assigning to `*s->v` modifies it through the pointer.

## Functions

### Export

For each function identifier `<f>` returning `<R>` with parameter list `<P>`, the translator generates a function-pointer member in the innermost struct. The struct hierarchy follows the same scheme as for variables:

```c
struct <path>_<name> { <R> (*<f>)(<P>); };
struct <path> { struct <path>_<name> <name>; } <path> = { .<name> = { <f> } };
```

Where a parcel exports both variables and functions, all members appear in the single innermost struct.

### Import

The import file follows the same form as for variables. The importer calls the function as `<stem>-><f>(<args>)`.

### Example

**`foo.c`** — declares and exports parcel `p` at path `foo`, exporting function `f`:

```c
#pragma parcel p { f }

void f(int x) { /* ... */ }

#include "export/foo/p"
```

**`_parcel/export/foo/p`** — generated by the translator:

```c
struct foo_p { void (*f)(int); };
struct foo { struct foo_p p; } foo = { .p = { f } };
```

**`bar.c`** — imports parcel `foo/p` with stem `s`:

```c
#include "import/foo/p.s"

s->f(42);
```

**`_parcel/import/foo/p.s`** — generated by the translator:

```c
struct foo_p { void (*f)(int); };
extern struct foo { struct foo_p p; } foo;
struct foo_p *s = &foo.p;
```

`s->f(42)` in `bar.c` calls `f` as defined in `foo.c` through the function pointer.

## Criticisms

**1. Cross-parcel dependency makes independent generation impossible.**
The global struct variable (named after the first path segment) must contain a member for every parcel in that path. If parcels `foo/p` and `foo/q` both exist, both export files must contribute to a single `struct foo` with members `p` and `q`. But each export file is generated in isolation: `export/foo/p` does not know about `foo/q` and cannot include a `q` member. The translator must scan all parcels at every path level before generating any export file at that level — a global dependency that prevents incremental or independent generation and significantly complicates the translation architecture.

**2. Path-segment name collides with exported identifier names.**
The global struct variable is named after the first path segment. If that segment matches any exported identifier in the same file, two global variables with the same name are declared in the same translation unit. In the `hello_world` example, path `output` with parcel `stdout` exporting identifier `output` would produce both `out_Output output = print;` and `struct output { ... } output = { ... };` in the same file — a direct identifier collision and a compilation error. The underscore approach avoids this because its canonical struct name (`output_stdout`) is never a bare path segment.

**3. Struct definition duplicated across export and import files.**
Both the innermost struct type and the enclosing path struct type must be re-declared in the import file, duplicating the full chain of definitions. As with the underscore approach, including both export and import files in the same translation unit would cause duplicate struct definition errors. The problem is compounded here because more struct types are involved at each nesting level.

**4. Variable access syntax is asymmetric with the README, and function-pointer variables retain the extra-indirection inconsistency.**
These are identical to the criticisms of the underscore approach. Variables require `*stem->id` rather than the `stem->id` form described in the README, and function-pointer typed variables (such as `output` in `hello_world`) require an extra dereference to call (`(*s->output)(greeting)`) rather than the `s->output(greeting)` form shown in the README.

**5. Exported identifiers must have static storage duration.**
As with the underscore approach, the struct initialiser requires all referenced identifiers to have static storage duration. Local or `auto` variables cannot be exported; the translator should diagnose this case.

**6. No support for `const`-qualified exports.**
All variable members are non-const pointers, as in the underscore approach. A `const` variable exported via `T *` in the struct loses its qualifier.

## Contrast with the Underscore-Path Approach

| | Underscore-path | Nested struct |
|---|---|---|
| Canonical struct variable name | `foo_p` (flat) | `foo.p` (nested within `foo`) |
| Innermost struct tag | `foo_p` | `foo_p` (identical) |
| Stem pointer type | `struct foo_p *` | `struct foo_p *` (identical) |
| Stem pointer initialiser | `&foo_p` | `&foo.p` |
| Importer access syntax | `*s->v`, `s->f()` | `*s->v`, `s->f()` (identical) |
| Global namespace entries | One per parcel (`foo_p`) | One per first path segment (`foo`) |
| Cross-parcel dependency | None; each parcel independent | All parcels in same path share `struct foo` |
| Generation | Fully incremental | Requires full path-level aggregation |
| Path-segment/identifier collision | Not possible | Fatal when first segment = exported identifier |
| Struct definitions in import file | Two (innermost + extern) | Two per nesting level |

The two approaches are identical from the importer's perspective: the stem pointer type, the access syntax, and the semantics of `*s->v` and `s->f()` are unchanged. The difference is entirely in how the canonical struct variable is organized globally. The nested approach reduces the number of top-level global identifiers (one per path root rather than one per parcel) at the cost of requiring the translator to aggregate parcels across files and risking collision between the path-root variable and exported identifier names.

For single-segment paths — the common case in practice — the nested struct adds exactly one extra level of wrapping (`struct foo { struct foo_p p; }`) with no semantic benefit over the flat `struct foo_p foo_p` of the underscore approach. The underscore approach is simpler to generate, simpler to reason about, and free of the cross-parcel aggregation problem.
