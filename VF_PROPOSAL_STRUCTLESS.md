# Variable and Function Proposal: Long Identifiers (Structless)

This proposal exports each variable and function identifier independently under its own canonical global name, with no struct grouping. The canonical naming scheme is identical across all identifier categories — typedef, variable, and function — producing a uniform namespace.

## Variables

### Canonical Name

Each exported variable identifier is assigned a canonical name following the same scheme used for typedefs:

```
<path>/<name> { <id> }  →  <path_underscored>_<name>_<id>
```

For example, path `foo`, parcel name `p`, identifier `v` produces the canonical name `foo_p_v`. The export file defines a global pointer variable of this name, initialised with the address of the exported variable. The import file defines a stem-prefixed pointer variable assigned the value of the canonical pointer.

### Export

For each variable identifier `<id>` of type `<T>`, the translator generates a pointer-typed global variable in `export/<path>/<name>`, initialised with the address of the exported variable:

```c
<T> *<path_>_<name>_<id> = &<id>;
```

The export file must be included _after_ the variable definition it references.

### Import

For each variable identifier `<id>` of type `<T>`, the translator generates a pointer-typed global variable in `import/<path>/<name>.<stem>`, assigned the value of the canonical export variable:

```c
extern <T> *<path_>_<name>_<id>;
<T> *<stem>_<id> = <path_>_<name>_<id>;
```

The importer reads the variable as `*<stem>_<id>` and writes to it as `*<stem>_<id> = <value>`.

### Example

**`foo.c`** — declares and exports parcel `p` at path `foo`, exporting variable `v`:

```c
#pragma parcel p { v }

int v = 0;

#include "export/foo/p"
```

**`_parcel/export/foo/p`** — generated by the translator:

```c
int *foo_p_v = &v;
```

**`bar.c`** — imports parcel `foo/p` with stem `s`:

```c
#include "import/foo/p.s"

int x = *s_v;
*s_v = 42;
```

**`_parcel/import/foo/p.s`** — generated by the translator:

```c
extern int *foo_p_v;
int *s_v = foo_p_v;
```

`*s_v` reads the current value of `v` defined in `foo.c`; assigning to `*s_v` modifies it through the pointer.

## Functions

### Export

For each function identifier `<f>` with return type `<R>` and parameter list `<P>`, the translator generates a function-pointer variable in `export/<path>/<name>`, initialised with the function:

```c
<R> (*<path_>_<name>_<f>)(<P>) = <f>;
```

The canonical name follows the same scheme as for variables. The export file must be included _after_ the function definition it references.

### Import

For each function identifier `<f>`, the translator generates a function-pointer variable in `import/<path>/<name>.<stem>`, assigned the value of the canonical export variable:

```c
extern <R> (*<path_>_<name>_<f>)(<P>);
<R> (*<stem>_<f>)(<P>) = <path_>_<name>_<f>;
```

The importer calls the function as `<stem>_<f>(<args>)`.

### Example

**`foo.c`** — declares and exports parcel `p` at path `foo`, exporting function `f`:

```c
#pragma parcel p { f }

void f(int x) { /* ... */ }

#include "export/foo/p"
```

**`_parcel/export/foo/p`** — generated by the translator:

```c
void (*foo_p_f)(int) = f;
```

**`bar.c`** — imports parcel `foo/p` with stem `s`:

```c
#include "import/foo/p.s"

s_f(42);
```

**`_parcel/import/foo/p.s`** — generated by the translator:

```c
extern void (*foo_p_f)(int);
void (*s_f)(int) = foo_p_f;
```

`s_f(42)` in `bar.c` calls `f` as defined in `foo.c` through the function pointer.

## Criticisms

**1. Import initialiser is not valid in standard C (critical).**
Both `int *s_v = foo_p_v;` and `void (*s_f)(int) = foo_p_f;` are file-scope (static storage duration) variable definitions. The C standard requires that initialisers for objects with static storage duration be constant expressions. The value of a global pointer variable — even one that holds an address constant — is not a constant expression: `foo_p_v` and `foo_p_f` are variables, not compile-time constants. The proposed import files are therefore invalid in standard C. The struct-based approaches avoid this entirely by initialising the stem pointer with `&foo_p`, the address of the canonical struct variable, which is an address constant and thus a valid static initialiser.

A workaround exists — using a macro in place of a variable declaration:

```c
extern int *foo_p_v;
#define s_v (*foo_p_v)
```

This is valid and gives natural read/write syntax (`s_v = 42`), but departs from the proposal's description of a _variable_ definition and introduces the usual macro hazards.

**2. "Function type" is not a valid variable type in C.**
The proposal describes the export-defined intermediate for function identifiers as "merely a function type." C does not permit variables of function type — only of function pointer type. The export file correctly uses `void (*foo_p_f)(int)` (a function pointer), but the proposal's description is imprecise and could mislead an implementor.

**3. Access syntax is asymmetric between variables and functions.**
Imported variables require an explicit pointer dereference (`*s_v`) while imported functions are called directly through the function pointer (`s_f(args)`). These are different idioms for identifiers that the parcel treats as peers. A caller must know which category each imported identifier belongs to in order to write correct access syntax.

**4. Inconsistency with the README's `stem->identifier` access idiom.**
The Parcel README describes variable and function access using the `stem->identifier` form, implying a struct pointer. This proposal produces `*s_v` and `s_f()` — neither matches that idiom. The `hello_world` example's `OUT->output(greeting)` would become `(*std_output)(greeting)` under this scheme, which is both inconsistent with the documented syntax and requires the caller to know that `output` is a function-pointer typed variable rather than a plain function.

**5. Global namespace populated with one name per exported identifier.**
Each exported identifier introduces an independent canonical global variable (`foo_p_v`, `foo_p_f`, …). A parcel with _n_ identifiers adds _n_ names to the global namespace. The struct-based approaches add a single name per parcel. This increases the risk of collisions with user-defined identifiers and with canonical names from other parcels.

**6. No parcel handle type.**
The struct-based approaches produce a struct pointer (`struct foo_p *`) that groups all exported identifiers under a single stem. This handle can be passed to a function, stored, or returned — the full parcel interface travels as one value. This proposal has no equivalent: `s_v` and `s_f` are unrelated globals. Passing a parcel to a function requires passing each identifier separately as individual pointer arguments.

**7. Uniform `stem_id` naming conflates type aliases and pointer variables.**
The import-side naming convention `<stem>_<Id>` is shared with the typedef import convention. In a file that imports both typedefs and variables from the same parcel, `s_T` is a type alias (from a typedef import) and `s_v` is a pointer variable (from a variable import). These are syntactically indistinguishable at the import-include level and require the reader to consult the parcel declaration to determine which category each identifier belongs to. The struct approaches make the distinction explicit: `s_T` is always a type, while `s->identifier` always involves a pointer dereference.

## Contrast with Struct-Based Approaches

| | Structless (this proposal) | Underscore struct | Nested struct |
|---|---|---|---|
| Export global name(s) | One per identifier (`foo_p_v`, `foo_p_f`) | One per parcel (`foo_p`) | One per path root (`foo`) |
| Import initialiser | Value copy of pointer variable — **invalid static initialiser** | `&foo_p` — valid address constant | `&foo.p` — valid address constant |
| Variable access | `*s_v` | `*s->v` | `*s->v` |
| Function call | `s_f(args)` | `s->f(args)` | `s->f(args)` |
| Unified access idiom | None | `s->identifier` (with `*` for variables) | `s->identifier` (with `*` for variables) |
| Parcel handle type | None | `struct foo_p *` | `struct foo_p *` |
| Struct type duplication | None | Export + import files | Export + import files, per nesting level |
| Cross-parcel dependency | None | None | Requires aggregation per path root |
| Naming consistency with typedefs | Identical scheme (`path_name_id`) | Different (`path_name` for struct) | Different |
| Path-segment/identifier collision | None | None | Fatal when first segment = identifier name |

The structless approach is the only one whose canonical naming scheme is fully uniform across all three identifier categories. It also avoids struct type duplication and cross-parcel dependencies. These are genuine advantages. However, the invalid static initialiser is a fundamental defect in standard C that invalidates the scheme as stated. Resolving it requires either replacing the import variable definition with a macro, adding a level of pointer indirection (making the import variable `T **s_v = &foo_p_v`), or abandoning file-scope initialisation — each of which departs materially from the proposal and reintroduces complexity that partially offsets the structless simplicity.
